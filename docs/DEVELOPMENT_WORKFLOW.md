# 开发流程

本文档定义本项目的标准开发流程。后续所有开发、调试、修复和扩展都必须遵守该流程。

## 一、开发总原则

项目按开源软件方式维护，要求：

- 需求清楚。
- 技术路线清楚。
- 文档和代码同步。
- 模块边界清楚。
- 状态可恢复。
- 错误可追踪。
- 后续可扩展。

## 二、标准流程

每次开发必须按以下顺序执行：

1. 确认需求。
2. 确认技术路线。
3. 编写或修改代码。
4. 运行必要验证，确保修改具备可测试状态。
5. 提供明确启动命令，让用户自行启动软件并进行功能验收；除非用户明确要求，开发者或智能体不主动启动 GUI。
6. 用户确认功能无误后，再更新或补充相关文档。
7. 检查文档和代码是否一致。
8. 用户确认可以落库后，再执行 Git 暂存、提交和推送。
9. 汇报结果、验证、提交记录和风险。

## 三、需求确认

需求确认阶段必须回答：

- 本次要做什么。
- 本次不做什么。
- 输入是什么。
- 输出是什么。
- 影响哪些页面。
- 影响哪些配置。
- 影响哪些数据表。
- 影响哪些导出字段。
- 失败时如何处理。
- 是否需要用户进一步确认。

需求不明确时，不允许进入开发。

## 四、技术路线确认

技术路线确认阶段必须回答：

- 是否沿用 PySide6 和 PySide6-Fluent-Widgets。
- 是否沿用 SQLite。
- 是否影响 Selenium 或 Playwright。
- 是否新增依赖，尤其是是否和 `qfluentwidgets` 导入包名冲突。
- 是否影响打包迁移。
- 是否影响配置结构。
- 是否影响数据库结构。
- 是否影响文档结构。
- 是否影响 Conda 环境、依赖版本或驱动目录。

技术路线不明确时，不允许进入开发。

## 五、环境管理流程

环境配置必须遵守：

1. 使用 Anaconda。
2. Conda 环境名称固定为 `gmap`。
3. Python 版本优先使用 3.11，除非依赖兼容性要求调整。
4. pip 安装优先使用清华源。
5. 所有依赖版本必须写入 `requirements.txt`。
6. Conda 环境定义必须写入 `environment.yml`。
7. 浏览器驱动和自动化工具缓存尽量放在项目内 `drivers/` 目录。

如果环境创建或依赖安装失败，必须记录失败原因，并按实际错误决定是否需要用户确认。

## 六、文档同步

用户验收功能前，不要求完成最终文档落地和 Git 提交，避免未确认版本被提前固化。用户确认功能无误后，必须同步检查并更新相关文档：

- 需求变更先更新 `docs/REQUIREMENTS.md`。
- 技术设计变更先更新 `docs/DESIGN.md`。
- 目录结构变更先更新 `docs/PROJECT_STRUCTURE.md`。
- 开发流程变更先更新 `docs/DEVELOPMENT_WORKFLOW.md`。
- 用户使用方式变更先更新 `README.md`。
- 版本变化更新 `CHANGELOG.md`。

最终提交前必须保证文档和代码一致。用户明确要求立即更新的规范文件除外，例如 `AGENTS.md` 或本文件本身。

## 七、实现阶段

实现阶段要求：

- 遵守模块边界。
- 代码注释使用中文。
- 核心逻辑写清楚原因和边界。
- 不实现未确认需求。
- 不跨层硬耦合。
- 不把浏览器引擎细节写进 GUI。
- 不把数据库写入逻辑写进解析器。

## 八、验证阶段

根据修改类型运行对应验证：

| 修改类型 | 必要验证 |
| --- | --- |
| 配置读取 | 验证 JSON 解析和默认值 |
| 关键词生成 | 验证组合结果和 URL 编码 |
| 数据库 | 验证初始化、插入、去重、查询 |
| 导出 | 验证 CSV 和 Excel 生成 |
| GUI | 验证窗口启动、按钮事件、表格刷新 |
| 浏览器引擎 | 验证 Chrome 或 Edge 启动和打开 URL |
| 解析逻辑 | 验证可见 DOM 字段提取 |
| Windows 打包 | 验证 PyInstaller 构建成功、zip 生成、打包后 exe `--check` 正常 |

如果验证无法完成，必须说明原因。

## 九、Google Maps DOM 调试流程

Google Maps 页面结构可能变化。调试时必须遵守：

1. 先观察页面实际 DOM。
2. 尽量使用稳定属性和结构定位。
3. 如果无法定位目标元素，停止猜测并询问用户。
4. 如果用户愿意，可以让用户手动指出元素。
5. 不因为定位困难而擅自改为点击详情页。
6. 不因为定位困难而增加未确认的新采集策略。

## 十、异常处理流程

遇到异常时：

1. 记录异常上下文。
2. 判断是否是单个关键词失败。
3. 判断是否达到连续失败阈值。
4. 判断是否需要暂停整批任务。
5. 如果异常原因不清楚，询问用户。

必须记录的上下文：

- 当前关键词。
- 当前国家。
- 当前地区。
- 当前城市。
- 当前 URL。
- 当前浏览器。
- 当前引擎。
- 错误原因。
- 发生时间。

## 十一、完成标准

一项开发完成必须满足：

- 需求已实现。
- 未实现未确认需求。
- 相关验证已运行。
- 用户已完成验收或明确允许进入文档和提交阶段。
- 文档已同步。
- 已说明验证结果。
- 已说明遗留风险。

未满足以上条件时，不得声称完成。

## 十二、发布流程

发布 Windows 发行版时按以下顺序执行：

1. 确认版本号和发布范围。
2. 更新 `CHANGELOG.md`，把本次发布内容落到明确版本和日期。
3. 确认 `README.md` 中的安装、启动和发行版说明与实际产物一致。
4. 运行单元测试。
5. 执行 `.\scripts\build_windows_release.ps1 -Version <版本号>`。
6. 使用生成目录内的 `GoogleMaps_LeadForge.exe --check` 验证打包入口。
7. 检查 `dist/GoogleMaps_LeadForge-v<版本号>-windows-x64.zip` 存在。
8. 提交代码，创建版本 tag，推送到 GitHub。
9. 创建 GitHub Release，并上传对应 zip 包。

Release 说明应以 `CHANGELOG.md` 中对应版本内容为准。发布过程不允许把 GitHub token 写入仓库、文档、日志或命令历史中。
