# 需求说明

本文档记录项目当前已经确认的需求边界。后续任何新增、删除或修改需求，都必须同步更新本文档。

## 一、项目背景

用户需要一个带图形界面的 Python 桌面软件，通过组合国家、地区、城市和行业关键词，批量生成 Google Maps 搜索链接，并从搜索结果列表页采集商家基础信息。

项目后续会持续扩展，因此第一版必须重视：

- 健壮性。
- 稳定性。
- 可恢复性。
- 可扩展性。
- 文档完整性。
- 目录结构清晰。

## 二、核心目标

第一版软件需要完成以下目标：

1. 从配置文件加载国家、地区和城市。
2. 在 GUI 中通过复选框选择国家和地区。
3. 选择地区后默认包含该地区下全部城市。
4. 在 GUI 中输入行业关键词，一行一个。
5. 批量生成 Google Maps 搜索链接。
6. 使用可视化浏览器顺序访问搜索链接。
7. 只滚动搜索结果列表页，不进入商家详情页。
8. 从可见 DOM 中解析商家基础字段。
9. 使用 SQLite 保存任务状态和商家记录。
10. 按 Google Maps 链接全局去重。
11. 支持暂停、继续、停止和失败重试。
12. 支持关闭软件后恢复上次任务。
13. 支持连续失败达到阈值后自动暂停。
14. 支持导出 CSV 和 Excel。
15. 支持从某个 Google Maps 任务批次创建“官网探索”二次任务。
16. 官网探索模块必须能按商家是否存在官网自动分流：有官网进入待执行队列，无官网直接跳过。
17. 结果管理页必须展示并维护官网探索字段，作为最终汇总视图。
18. 软件运行、官网探索、结果查看和导出都必须以 Task 为基本单位。
19. 官网探索必须支持静态请求、正则提取和浏览器兜底，避免单个官网请求失败或卡住导致整批任务无法继续。
20. 项目必须提供 Windows exe 发行版，普通用户可下载压缩包后直接运行，不要求安装 Python 环境。

## 三、关键词生成规则

第一版不包含街道级搜索。搜索词组合规则为：

```text
行业关键词 + in + 城市, 地区, 国家
```

示例：

```text
Car Wrap Shop in Biberach, Baden-Wuerttemberg, Germany
```

生成的 Google Maps URL 示例：

```text
https://www.google.com/maps/search/Car+Wrap+Shop+in+Biberach%2C+Baden-Wuerttemberg%2C+Germany
```

## 四、地区配置需求

地区配置从 JSON 文件读取。

第一版包含德国测试配置，至少包含 10 个地区，并为每个地区提供若干城市用于测试。

配置字段包括：

- `name`：中文显示名称。
- `search_name`：用于搜索链接的英文或目标语言名称。
- `regions`：地区列表。
- `cities`：城市列表。

## 五、运行配置需求

运行配置从 `config/app_config.json` 读取，至少包含：

- 默认浏览器。
- 默认浏览器引擎。
- 页面初始停留秒数。
- 每个关键词完成后的随机停留范围。
- 滚动加载间隔随机范围。
- 连续无新增停止次数。
- 最大滚动次数。
- 页面加载超时时间。
- 连续失败自动暂停阈值。
- 导出目录。
- 日志目录。
- 数据库路径。
- 浏览器驱动目录。
- Playwright 浏览器缓存目录。

## 六、采集字段

第一版采集字段如下：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 商家名称 | 是 | 从列表页解析 |
| 地址 | 否 | 列表页能解析到则保存，解析不到留空 |
| 电话 | 否 | 列表页能解析到则保存，解析不到留空 |
| 官网 | 否 | 列表页能解析到则保存，解析不到留空 |
| 评分 | 否 | 从列表页解析 |
| 评论数量 | 否 | 从列表页解析 |
| 商家分类 | 否 | 从列表页解析 |
| Google Maps 链接 | 是 | 全局去重依据 |
| 来源关键词 | 是 | 多个行业关键词命中时用英文逗号合并 |

## 七、采集流程需求

每个关键词任务执行流程：

1. 根据关键词生成 Google Maps 搜索链接。
2. 打开可视化浏览器窗口。
3. 访问搜索链接。
4. 等待页面初始加载。
5. 定位搜索结果列表。
6. 解析当前已经加载的商家卡片。
7. 向下滚动列表。
8. 按配置随机停留。
9. 再次解析新增商家。
10. 如果到达列表底部，当前关键词完成。
11. 如果连续多次滚动没有新增商家，当前关键词完成。
12. 如果超时或解析异常，当前关键词失败。
13. 单个关键词失败后继续下一个关键词。
14. 连续失败达到配置阈值时，暂停整批任务。

## 八、暂停和恢复需求

软件必须支持：

- 手动暂停任务。
- 暂停后继续任务。
- 停止任务。
- 关闭软件后重新打开并恢复未完成任务。

任务状态必须持久化到 SQLite。

创建任务批次时，任务配置页中的浏览器、页面等待、滚动间隔、最大滚动次数、连续失败阈值等运行参数必须作为本次任务快照保存到 SQLite。暂停、继续、失败重试和软件重启后的恢复都应继续使用该快照。

关键词状态包括：

- 待执行。
- 执行中。
- 完成。
- 失败。
- 已跳过。

## 九、失败处理需求

失败处理规则：

- 单个关键词失败时记录失败原因。
- 单个关键词失败后继续下一个关键词。
- 连续失败次数达到配置阈值时暂停整批任务。
- 任务结束后支持重试失败关键词。
- 遇到验证码、异常页面或疑似风控页面时暂停任务并等待人工处理。

## 十、去重和导出需求

去重规则：

- 商家主记录按 Google Maps 链接全局去重。
- 同一链接被多个行业关键词命中时，合并来源关键词。
- 来源关键词字段只保存用户输入的行业关键词。
- 完整搜索词可能包含城市、地区和逗号，因此存入命中关系表，不直接写入来源关键词字段。
- 来源关键词使用英文逗号分隔，例如 `Car Wrap Shop,PPF`。

导出规则：

- 导出必须从 SQLite 中的去重结果读取。
- 导出应按用户在结果管理页选择的 Task 过滤结果。
- 支持 CSV。
- 支持 Excel。
- 导出字段必须和结果管理页表格字段一致。

## 十一、GUI 页面需求

第一版 GUI 使用 PySide6 和 PySide6-Fluent-Widgets 构建，并使用左侧侧边导航：

1. 任务配置。
2. 任务执行。
3. 官网探索。
4. 结果管理。
5. 设置。

左侧导航固定展示页面入口，右侧展示当前页面内容。设置入口固定在左侧导航底部，展开宽度保持紧凑，默认窗口尺寸不应过大。后续新增模块时，应优先增加左侧导航项，而不是新增独立窗口。

详细页面设计见 `README.md` 和 `docs/DESIGN.md`。

GUI 风格要求：

- 使用 Fluent 风格组件。
- 页面布局清晰、稳定、适合长期工具使用。
- 优先使用成熟控件实现按钮、复选框、输入框、表格、提示、导航、设置卡片和主题切换。
- 页面应使用 Fluent 卡片化分组承载相关功能，避免功能项拥挤或混杂。
- 设置页只展示外观、全局运行参数、项目路径和维护操作，不展示项目开发文档入口。
- 数字输入框应避免鼠标滚轮误触修改运行参数。
- 不使用 Tkinter。
- 不使用 Web 前端框架。

任务执行页状态反馈要求：

- 任务执行页必须提供当前 Task 选择器，用户可以选择要执行的任务。
- 任务运行中必须锁定 Task 选择器，避免前端显示任务与后台实际运行任务不一致。
- 点击开始后必须立即显示“启动中”，避免浏览器启动期间用户看不到反馈。
- 任务启动或运行中必须显示不确定进度条。
- 运行过程中必须显示当前关键词、国家、地区、城市、浏览器引擎、原始命中数量和去重商家数量。
- 关键词状态变化后应刷新关键词队列表和状态面板。

官网探索页要求：

- 允许从某个已存在的 Google Maps 任务批次创建官网探索批次。
- 自动读取该批次命中的去重商家。
- 有官网的商家进入待执行状态。
- 没有官网的商家直接标记为跳过。
- 一个 Google Maps 来源任务可以创建多个官网探索批次，探索批次必须和来源任务绑定。
- 来源任务切换后，只展示该来源任务下的官网探索批次。
- 官网探索页必须展示当前探索批次的完成、失败、跳过、待探索、当前商家和当前官网。
- 支持配置探索深度、每站最多页面数和请求超时秒数。
- 优先使用静态请求获取官网 HTML。
- 静态请求失败、请求超时或没有提取到核心联系方式时，必须尝试浏览器自动化兜底。
- 静态请求和浏览器兜底都失败时，必须标记当前官网探索任务失败并记录原因，不允许长期停留在执行中。
- 静态请求失败但浏览器兜底成功时，最终状态应按成功统计，避免前端状态和数据库状态不一致。
- 结果管理页必须能看到官网探索字段，作为最终汇总结果的一部分。

辅助脚本要求：

- 提供登录浏览器脚本，直接启动真实 Chrome/Edge 进程，并使用与采集任务相同的浏览器用户数据目录，便于用户先登录 Google 账户。
- 提供运行产物清理脚本，默认清理本地数据库、日志、导出和调试输出，但不得删除关键词输入、配置文件或浏览器登录缓存。
- 提供设置页清理按钮，允许用户二次确认后清理数据库、日志、导出、调试输出和浏览器登录缓存，使项目回到接近全新测试状态。
- 提供 Windows 发行版构建脚本，生成包含 exe、默认配置、运行目录、README、LICENSE 和 CHANGELOG 的 zip 包。

## 十二、不做的内容

第一版不做：

- 街道级搜索。
- 多浏览器并发。
- 无头模式。
- 自动绕过验证码或风控。
- 登录账号管理。
- 点击商家详情页补全字段。
- 云端同步。
- 人工备注编辑。

说明：Playwright 作为后续可插拔引擎保留在配置和界面中，但当前第一版实际执行采集只支持 Selenium。选择 Playwright 启动任务时应明确提示暂不支持，不应静默切换到其他引擎。

## 十三、用户特别约束

开发过程中必须遵守：

1. 如果遇到任何问题或无法解释清楚的地方，随时询问用户。
2. 调试 Google Maps DOM 时，如果找不到元素或无法定位 DOM，询问用户是否协助人工查找。
3. 不做用户没有提到或没有确认的其他需求。

## 十四、开发环境需求

开发环境必须满足：

1. 使用 Anaconda。
2. 创建名为 `gmap` 的 Conda 虚拟环境。
3. Python 版本由开发者按依赖兼容性选择，当前设计优先使用 Python 3.11。
4. 所有项目依赖安装在 `gmap` 环境中。
5. pip 安装优先使用清华源 `https://pypi.tuna.tsinghua.edu.cn/simple`。
6. 自动化库需要的驱动、浏览器缓存和开发工具应尽量放在当前项目目录中。
7. `requirements.txt` 必须明确标注依赖版本。
8. `environment.yml` 必须和实际环境保持一致。

## 十五、发行版需求

Windows 发行版要求：

1. 使用 PyInstaller onedir 模式构建。
2. exe 名称为 `GoogleMaps_LeadForge.exe`。
3. 发行包命名格式为 `GoogleMaps_LeadForge-v<版本号>-windows-x64.zip`。
4. 发行包解压后，配置文件、数据库、导出、日志和浏览器缓存都位于 exe 同级目录。
5. 发行版不内置 Chrome 或 Edge，用户电脑仍需安装至少一种受支持浏览器。
6. 发行版不改变验证码、登录限制、访问限制或风控处理策略。
7. 发布前必须验证源码入口 `--check`、单元测试和打包后 exe `--check`。
