# 智能体开发规范

本文档面向后续参与本项目的 Codex、AI 智能体和人工开发者。所有开发活动必须遵守本文档中的规则。除非用户明确修改规范，否则本文档优先作为项目内开发约束。

## 基本原则

1. 所有项目回复、项目文档、代码注释默认使用中文。
2. 第三方库名、API 名称、命令、错误信息、协议字段可以保留英文。
3. 不允许在需求未确认时直接开发业务代码。
4. 不允许在技术路线未确认时擅自选择新的框架、引擎或架构。
5. 不允许实现用户没有提出或没有确认的额外需求。
6. 不允许自动绕过验证码、登录限制、访问限制或其他风控机制。
7. 遇到无法解释清楚的问题，必须向用户确认，不得自作主张。
8. 调试 Google Maps DOM 时，如果无法定位目标元素，必须询问用户是否协助人工定位，不得随意更换采集策略。

## 固定开发流程

每次新增功能、修改功能、修复问题或调整架构时，必须遵循以下流程：

1. 确认需求。
2. 确认技术路线。
3. 编写或修改代码。
4. 运行必要验证，确保修改具备可测试状态。
5. 启动软件或提供明确运行方式，让用户先进行功能验收。
6. 用户确认功能无误后，再补齐 `README.md`、`docs/`、`CHANGELOG.md`、项目结构说明等相关文档。
7. 再次检查文档是否和代码一致。
8. 用户确认可以落库后，再执行 Git 暂存、提交和推送。
9. 汇报修改内容、验证结果、提交记录和遗留风险。

如果需求、技术路线或边界不清晰，必须暂停开发并向用户提问。

严禁在用户尚未验收功能前，直接完成全部文档落地并提交 Git。用户明确要求立即更新的规范文件除外，例如本文件本身。

## 需求确认规范

需求确认时必须明确以下内容：

- 本次修改要解决什么问题。
- 本次修改不包含什么内容。
- 输入是什么。
- 输出是什么。
- 数据如何保存。
- 失败时如何处理。
- 是否影响现有配置文件。
- 是否影响数据库结构。
- 是否影响 GUI 页面。
- 是否影响 PySide6-Fluent-Widgets 组件使用方式。
- 是否影响导出字段。
- 是否需要更新测试。

对于任何可能扩大范围的内容，必须先询问用户。

## 技术路线确认规范

技术路线确认时必须明确以下内容：

- 是否沿用当前技术栈。
- 是否新增第三方依赖。
- 是否和 `PySide6-Fluent-Widgets` 或 `qfluentwidgets` 导入包名冲突。
- 是否影响打包和迁移。
- 是否影响 Selenium 或 Playwright 引擎接口。
- 是否影响 SQLite 数据模型。
- 是否影响配置文件结构。
- 是否影响文档结构。
- 是否影响 Conda 环境、依赖版本、镜像源或驱动目录。

未经用户确认，不得引入新的大型框架、数据库、服务端组件或云服务。

## 环境管理规范

1. 使用 Anaconda 管理环境。
2. Conda 环境名称固定为 `gmap`。
3. Python 版本优先使用 3.11，除非依赖兼容性要求调整。
4. pip 安装优先使用清华源 `https://pypi.tuna.tsinghua.edu.cn/simple`。
5. 所有 Python 依赖必须写入 `requirements.txt`，并标注明确版本。
6. Conda 环境必须写入 `environment.yml`。
7. Selenium、Playwright 或其他自动化工具需要的驱动和缓存应尽量放在项目内 `drivers/` 目录。
8. 不得把驱动和工具散落到系统临时目录后不做记录。

## 文档同步规范

每次开发都必须检查文档是否需要更新。文档和代码必须完全对应。

必须更新文档的情况：

- 新增、删除或移动目录。
- 新增、删除或重命名代码文件。
- 新增或删除 GUI 页面、按钮、表格、字段。
- 修改采集字段。
- 修改任务状态流转。
- 修改数据库表结构。
- 修改配置字段。
- 修改导出格式。
- 修改运行命令。
- 修改异常处理策略。
- 修改浏览器引擎策略。

对应更新规则：

- 项目说明、安装运行、使用方式变化，更新 `README.md`。
- 需求变化，更新 `docs/REQUIREMENTS.md`。
- 技术设计变化，更新 `docs/DESIGN.md`。
- 目录和文件职责变化，更新 `docs/PROJECT_STRUCTURE.md`。
- 开发流程变化，更新 `docs/DEVELOPMENT_WORKFLOW.md`。
- 版本变化，更新 `CHANGELOG.md`。
- 智能体规则变化，更新 `AGENTS.md`。

禁止出现代码已经变化但文档仍描述旧行为的情况。

## 代码注释规范

1. 代码注释必须使用中文。
2. 核心类必须说明职责、依赖和边界。
3. 核心函数必须说明输入、输出、异常和副作用。
4. 复杂流程必须说明设计原因。
5. 浏览器自动化逻辑必须说明定位依据和失败处理。
6. 数据库写入逻辑必须说明去重规则和事务边界。
7. 配置解析逻辑必须说明默认值和兼容策略。
8. 不写只重复代码含义的低价值注释。

示例：

```python
# 根据 Google Maps 链接进行全局去重。
# 如果同一商家被多个关键词命中，只合并来源关键词，不重复插入商家主记录。
```

## 架构边界规范

项目必须保持清晰分层：

- GUI 层只负责界面展示、用户输入和信号连接。
- GUI 层必须优先使用 PySide6-Fluent-Widgets 组件，底层保持 PySide6。
- 任务调度层负责任务队列、暂停、继续、失败重试和状态流转。
- 浏览器引擎层负责 Selenium 和 Playwright 的统一封装。
- 解析层负责从可见 DOM 中提取商家字段。
- 存储层负责 SQLite 表结构、事务、去重和查询。
- 导出层负责 CSV 和 Excel 输出。
- 配置层负责读取、校验和保存 JSON 配置。
- 日志层负责运行日志和调试日志。
- 公共层负责通用类型、异常、时间和路径工具。

禁止跨层硬编码调用。例如 GUI 层不得直接操作 Selenium DOM，解析层不得直接写 SQLite。

## 浏览器自动化规范

1. 第一版只使用可视化浏览器窗口。
2. 第一版只使用单进程、单浏览器窗口。
3. 默认浏览器是 Chrome，同时支持 Edge。
4. Selenium 和 Playwright 必须通过统一接口接入。
5. 不点击商家详情页。
6. 只解析搜索结果列表页已经加载出的可见 DOM。
7. 页面加载、滚动停留、关键词间隔都必须使用配置文件控制。
8. 到达列表底部或连续多次滚动无新增商家时，当前关键词完成。
9. 遇到验证码、异常页面或疑似风控页面，应暂停任务并等待人工处理。

## 数据规范

商家主记录按 Google Maps 链接全局去重。

同一 Google Maps 链接被多个关键词命中时：

- 不重复插入商家主记录。
- 合并来源行业关键词。
- 来源关键词字段只保存用户输入的行业关键词。
- 完整搜索词保存到命中关系表，不直接写入来源关键词字段。
- 来源关键词用英文逗号分隔。
- 更新时间刷新为最后一次采集时间。

导出必须从 SQLite 中的去重结果读取，不允许从临时内存列表直接导出。

## 配置规范

配置文件分为两类：

- `config/locations.de.json`：国家、地区和城市配置。
- `config/app_config.json`：运行配置。

配置文件必须可读、可迁移、可手动修改。新增配置字段时必须提供默认值，并更新 README 和设计文档。

## 错误处理规范

错误必须被记录，并尽量保留上下文：

- 当前关键词。
- 当前国家。
- 当前地区。
- 当前城市。
- 当前 URL。
- 当前浏览器。
- 当前引擎。
- 错误类型。
- 错误原因。
- 发生时间。

单个关键词失败后，应标记失败并继续下一个关键词。连续失败达到配置阈值时，必须暂停整批任务。

## 测试和验证规范

每次代码修改后至少执行与修改相关的验证：

- 配置解析修改后，验证 JSON 能被正确读取。
- 数据库修改后，验证表初始化、插入、去重、查询。
- 导出修改后，验证 CSV 和 Excel 文件能生成。
- GUI 修改后，验证界面能启动，按钮和表格不报错。
- GUI 依赖修改后，验证 `qfluentwidgets` 能正常导入。
- 浏览器引擎修改后，验证 Chrome 或 Edge 能被打开。

如果由于环境、网络、浏览器或 Google Maps 页面变化无法完成验证，必须在最终说明中明确写出。

## 禁止事项

禁止以下行为：

- 未确认需求直接开发。
- 未确认技术路线擅自换框架。
- 未确认就把 GUI 框架替换为 Tkinter、Web 前端或其他框架。
- 为了方便调试而删除用户数据。
- 自动绕过验证码或风控。
- 隐式修改配置文件结构。
- 修改导出字段但不更新文档。
- 修改数据库结构但不更新设计文档。
- 把 Selenium 或 Playwright 的细节散落到 GUI 层。
- 在没有用户确认的情况下增加新的业务需求。

## 遇到不确定情况时的处理方式

如果出现以下情况，必须停止当前决策并询问用户：

- Google Maps DOM 结构无法稳定定位。
- 字段解析结果和用户预期不一致。
- 某个字段需要点击详情页才能获得。
- 页面出现验证码、登录提示或异常风控页面。
- 配置文件字段需要调整。
- 数据库表结构需要变更。
- GUI 页面交互需要改变。
- 需要引入新依赖。
- 需要删除或迁移已有数据。

提问时必须说明：

- 当前遇到的问题。
- 已经确认的事实。
- 可选方案。
- 每个方案的影响。
- 推荐方案和理由。
